Subway Dash 2D PRD
一、專案概述
專案名稱： Subway Dash 2D

一句話描述： 一款基於網頁前端技術（HTML5 Canvas）打造的輕量級 2D 無盡跑酷街機遊戲，玩家透過跳躍與翻滾閃避障礙並收集金幣。

版本： v1.0 (MVP 最小可行性產品)

二、問題與目標
問題陳述
現代使用者在零碎時間需要能夠「隨點即玩」的輕量級娛樂。許多遊戲需要繁瑣的下載與安裝，或者依賴過重的遊戲引擎，導致網頁載入緩慢。

目標
技術驗證： 打造一個純粹使用 Vanilla JavaScript、HTML5 Canvas 和 Web Audio API 的遊戲，不依賴任何外部框架（如 Phaser 或 Unity），確保極速載入。

使用者體驗： 提供流暢的 60 FPS 遊戲體驗，並隨著遊玩時間自動增加難度（遊戲速度），帶來刺激的挑戰感。

跨平台支援： 透過整合鍵盤與觸控事件，確保 PC 與移動端網頁均可遊玩。

三、目標用戶
用戶特徵
喜歡休閒、快節奏街機遊戲的玩家。

希望在通勤、休息時間尋找短暫娛樂的行動裝置或桌面使用者。

用戶需求
極低的學習成本，操作直覺（只需跳與滾）。

即時的視覺（塗鴉風格、金幣閃光）與聽覺（合成器音效）回饋。

合理的難度曲線，遊戲需具備挑戰性以保持黏著度。

四、用戶故事
用戶故事1： 作為一名 PC 玩家，我想要 使用鍵盤（方向鍵/WASD/空白鍵）來控制角色跳躍和翻滾，以便 躲避不同高度的障礙物。

用戶故事2： 作為一名 手機玩家，我想要 透過螢幕上下滑動來控制角色，以便 在沒有實體鍵盤的設備上順暢遊玩。

用戶故事3： 作為一名 玩家，我想要 在遊戲中收集金幣，以便 看到分數累積並獲得成就感。

用戶故事4： 作為一名 玩家，我想要 在撞到障礙物時看到結算畫面並能快速重來，以便 我能立刻挑戰下一次的高分。

五、功能需求與驗收標準
功能1：遊戲核心循環與物理引擎 (Game Loop & Physics)
描述： 遊戲主迴圈（requestAnimationFrame），處理重力、跳躍物理、角色碰撞以及全局遊戲速度的動態提升。

驗收標準：

Given 遊戲正在進行中，When 幀數不斷增加，Then gameSpeed 必須以每幀 0.002 的速度緩慢增加。

Given 角色在空中，When 受到重力 (0.8) 影響，Then 角色的 Y 軸速度 (dy) 增加，直到接觸地面 (groundY)。

功能2：障礙物生成與判定系統 (Obstacle System)
描述： 根據遊戲速度，定時生成需要玩家應對的障礙物（高空招牌或地面路障）。

驗收標準：

Given 系統觸發生成條件，When 隨機數小於 0.3，Then 生成懸空的高處招牌（需翻滾躲避）；否則生成地面紅白路障（需跳躍躲避）。

Given 角色與障礙物的座標重疊，When 觸發矩形碰撞檢測，Then 遊戲結束並顯示 "BUSTED!" 畫面。

功能3：金幣與磁吸系統 (Coin System)
描述： 隨機生成在空中的金幣，並具有視覺旋轉特效。當玩家靠近時，金幣會自動吸附。

驗收標準：

Given 金幣生成在畫面上，When 玩家角色的中心點距離金幣小於 40 像素，Then 金幣被收集、分數 +1、播放清脆音效，並顯示 "+1" 噴漆飄字效果。

功能4：跨端控制系統 (Cross-platform Controls)
描述： 統一處理角色的「跳躍」與「翻滾」行為，並支援多種輸入裝置。

驗收標準：

Given 玩家按下 ↑、Space、W 或在螢幕向上滑動，When 角色在地面時，Then 賦予向上的跳躍力 (-16) 並播放音效。

Given 玩家按下 ↓、S 或在螢幕向下滑動，When 角色在地面時，Then 角色高度縮小為 35px，持續 40 幀；When 角色在空中時，Then 觸發快速下墜 (dy = 15)。

功能5：程序化音效 (Procedural Audio API)
描述： 不載入外部 MP3 檔案，純透過 Web Audio API 根據波形（Oscillator）即時生成音效。

驗收標準：

Given 角色跳躍，Then 播放頻率 300->600 的正弦波短音效。

Given 吃到金幣，Then 播放頻率 1200->1800 的三角波（Triangle）清脆音效。

Given 撞擊障礙物，Then 播放低頻鋸齒波（Sawtooth）的沉悶音效。

六、技術約束
必須遵守
前端實作僅能使用 Vanilla JavaScript、HTML5 Canvas API 以及 CSS3。

不依賴外部靜態圖片或音檔資源，畫面完全由 Canvas API 繪製，聲音完全由 Web Audio API 演算，以達成零延遲載入。

兼容性要求
外層容器 (#game-container) 使用固定尺寸 800x450，但必須透過 CSS Flexbox 確保在不同尺寸視窗下置中顯示。

必須同時監聽 keydown 與 touchstart/touchend 事件，保證雙端可用。

不要做
現階段不要引入後端資料庫或排行榜系統（僅保留當局分數）。

不要設計超過兩種以上的障礙物或道具系統，保持 MVP 的單純性。
